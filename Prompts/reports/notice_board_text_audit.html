1 · Label Clarity

Category Element / Label Issue Severity Recommendation
Label Clarity Modal field: Name Too generic — could mean a person's name, a group name, or a board title. Users must infer the intent. High Rename to Notice Board Name or Board Title to make the purpose unambiguous.
Label Clarity Column header: Group Name Inconsistent with the modal field labelled Name. Users editing a row see "Name" but the table shows "Group Name", causing a disconnect. Medium Align both to Notice Board Name or use a shared term like Board Name throughout.
Label Clarity Column header: S.No. Abbreviation "S.No." is not universally understood, especially by non-Indian audiences. Ambiguous to new users. Low Expand to # (minimal) or No. for clarity. If the audience is primarily Irish/international, use #.
Label Clarity Column header: Filter by (modal) Vague — does not explain what it filters. "Filter by what?" is the reader's natural question. Medium Rename to Recipient Type or Add Recipients by to communicate the intent clearly.
Label Clarity Main table column: Filter by Same issue as above. Displaying the filter mode in the table as "Filter by" does not tell the reader what the value means. Medium Rename column to Recipient Type. Values already make sense (Groups, Students, Teachers).
Label Clarity Chat Group Users table: Primary Contact/Staff Name The slash creates ambiguity — is this one person or two? The header conflates two semantically different roles. Medium Split into two columns: Primary Contact and Staff Name, or use a tooltip to clarify the dual meaning.
Label Clarity Legend item: Added in Chat group Lowercase "group" inconsistent with title-case convention used elsewhere. Minor but noticeable. Low Write as Added to Chat Group — title case and preposition corrected.
Label Clarity View Messages column: View Message read Users Mixed case, grammatically awkward phrasing. "read" is lowercase mid-sentence, and the word order is unusual. High Rename to Read Receipts or Readers for the column header. Show the count as a button label: View Readers (45).

2 · Terminology Consistency

Category Element / Label Issue Severity Recommendation
Terminology Toast: notice(s) vs Group message(s) Inconsistent use of "(s)" pluralisation pattern. Some toasts use it, some don't. It reads as awkward technical shorthand. Medium Use dynamic pluralisation in code: e.g. "2 notices published" / "1 notice published". Remove the "(s)" pattern entirely.
Terminology Modal source panel: Students List vs Teachers List The word "List" is appended in these panel titles but not in the main table header ("Group Name", not "Group Name List"). Inconsistent naming convention. Low Standardise: either always use "List" (e.g. Students List, Final List) or never (e.g. Students, Final Recipients).
Terminology Reply modal header vs button: User Reply column vs Can Reply modal column The table column is labelled "User Reply" but inside the modal it becomes "Can Reply". Two different labels for the same concept. High Pick one term. Can Reply is more descriptive. Use it consistently in both the main table column header and the modal.
Terminology Sidebar label: Notice Board vs page heading: Manage Notice Board The nav item just says "Notice Board" but the page title says "Manage Notice Board". The word "Manage" appears nowhere in the breadcrumb or nav, causing a cognitive mismatch. Low Either add "Manage" to the breadcrumb/nav, or remove it from the page heading. Consistent: Notice Board everywhere, or Manage Notice Board everywhere.
Terminology Status values: Published / Unpublished Status badges display only "P" or "U" — cryptic to unfamiliar users. Tooltip exists but is not obvious. Medium Display the full word in the badge: Published / Unpublished. Use colour coding to reinforce meaning. Remove single-letter abbreviation or make them optional via a compact/expanded view toggle.
Terminology Empty state: No data found. Generic fallback. Doesn't tell the user why data is empty or what action to take. Medium Replace with contextual messages like "Apply a filter above to load students" or "No students match the selected year and class."

3 · Typography Hierarchy

Category Element / Label Issue Severity Recommendation
Typography Modal subtitle: Please Fill Mandatory Fields Sentence case inconsistency — every word is capitalised (title case), but this is a sentence, not a title. Looks like a style error. Low Use sentence case: "Fields marked * are required" — also more informative and conventional.
Typography Filter sub-panel headers: Years and Class "Years" is plural; "Class" is singular. Inconsistent grammatical number for two parallel section headers. Low Standardise to plural: Years and Classes. Or singular: Year and Class.
Typography Panel headings: text-[17px] inline style The "Years" and "Class" sub-headers use a non-standard 17px size that sits between body text (14px) and section headings (base). Creates a weak hierarchy — they don't look like headings. Medium Use a consistent heading scale. Elevate these to font-semibold text-sm uppercase tracking-wide text-gray-500 to look like proper section labels rather than ambiguous large body text.
Typography Source list column headers: Student name (lowercase "n") Inconsistent capitalisation — other headers use Title Case (Primary Contact, Secondary Contact, Year, Class) but "Student name" uses sentence case. Low Change to Student Name to match the Title Case pattern of all adjacent column headers.
Typography Page heading: text-2xl font-semibold vs modal heading: text-xl font-semibold Two different heading weights/sizes for conceptually equivalent titles. Both use font-semibold but the page is 2xl and the modal is xl without a clear rationale documented in style. Low Fine as-is if intentional (modals are smaller contexts). Document or codify: modal titles = xl, page titles = 2xl. Apply consistently — the "Chat Group Users" heading is 2xl, same as the main page heading, making it feel equal in visual weight to a top-level page.

4 · Readability

Category Element / Label Issue Severity Recommendation
Readability Chat Users table: long column headers wrapping Headers like "Primary Contact/Staff Name" and "Secondary Contact Mobile" are very wide. At normal widths they wrap or truncate, harming scan-ability. Medium Shorten to Primary Contact and Sec. Contact in the header. Use full text in a tooltip. Or break the table into a card layout for this view.
Readability Loading row text: Loading more notices... / Loading more users... Good pattern, but the spinner and text share the same text-gray-500 colour, making them blend into the table body. Low contrast. Low Use a slightly more distinct treatment: a full-row light blue tint background and text-gray-700, so the loading state is visually distinct from data rows.
Readability Sidebar label: App Engagement Overview At 14px with standard weight in a narrow sidebar, this 3-word label wraps to 2 lines, pushing other nav items down unexpectedly. Low Shorten to App Engagement or Engagement Overview. Or increase sidebar width slightly to accommodate long labels cleanly.
Readability Groups Messages empty state: No results found. (italic) Italic text in a table cell is hard to read and looks like a placeholder rather than a real system message. Italics reduce readability, especially at small sizes. Low Remove italic. Use normal weight text-gray-500 with an optional icon: No messages to display.

5 · Button and Action Labels

Category Element / Label Issue Severity Recommendation
Button Labels Main toolbar: Add Single-word label gives no context about what is being added. "Add" what — a group? A notice? A user? High Rename to + New Notice Board or Add Notice Board. Mirrors industry convention (e.g. "+ New Post").
Button Labels Reply modal: don't allow to Reply Three problems: (1) starts with lowercase "d", (2) grammatically incorrect ("don't allow to" → should be "Don't Allow Reply"), (3) inconsistent style vs "Allow" and "Allow All". High Rename to Restrict Reply or Disallow Reply. Matches the positive counterpart "Allow Reply" / "Allow All" cleanly.
Button Labels Modal footer: Submit Generic. Doesn't communicate what is being submitted or what happens next. "Submit" is a form-era pattern that feels outdated. Medium Use context-aware labels: Save Notice Board (edit mode) and Create Notice Board (add mode).
Button Labels Modal source panel footer: Add to Final List Good label, but the ChevronRight icon implies navigation rather than an action of moving items. Can mislead users into thinking they'll be taken to another screen. Low Replace the ChevronRight with a → arrow or a plus icon. Or use a downward arrow since the Final List appears below the source list.
Button Labels Messages view: Apply Filter / Clear Filter Inconsistent capitalisation between these two adjacent buttons: "Apply Filter" (Title Case) vs "Clear Filter" (Title Case) — these are actually consistent, but compared with the modal's Apply Filter button (teal border) vs the main view's (no distinctive style), the pattern feels visually inconsistent. Low Unify button styling for "Apply Filter" across all views. Use the same teal border style for all filter-apply actions.
Button Labels Confirm dialog: Confirm Delete (title) + Delete (button) Redundant — "Confirm Delete" as a modal title and then "Delete" as the action button. Users read the title, then see "Delete" again without knowing what they're deleting. Medium Change the dialog title to Delete Notice Board? (includes the object). The confirm button can then simply say Delete — the object is clear from the title.

6 · Grammar and Spelling

Category Element / Label Issue Severity Recommendation
Grammar Reply modal header: Assign Can reply status to the Users Multiple issues: (1) "Can reply" mid-sentence should be lowercase, (2) "the Users" — article "the" is unnatural; should be "to Users", (3) whole phrase reads like a machine-translated label. High Rewrite as Manage Reply Permissions or Set Reply Access for Users.
Grammar Read/Unread modal header: Users Who Read & Doesn't read Messages Subject–verb agreement error: "Users … Doesn't" is grammatically incorrect (should be "don't"). Also, "read" is ambiguous (past vs present tense). High Rewrite as Message Read Status for the modal title. The tab labels ("Read" / "Unread") are sufficient to communicate the content.
Grammar Toast: Error: Group Name is required. "Group Name" is not what the field is labelled in the UI (it's labelled "Name"). Mismatch between field label and error message. Also, colon after "Error" is unusual in modern UI patterns. Medium Change to Notice Board Name is required. — matches the field label and removes the redundant "Error:" prefix (the red colour already communicates error).
Grammar Toast: Users list updated based on groups. Lowercase "list" inconsistent with title-case used for "Students List" in panel headers. Minor but part of a pattern. Low Rewrite as Students list updated based on selected groups. — adds "selected" for clarity and keeps lowercase consistent for body text.
Spelling Reply modal button: don't allow to Reply Starts with a lowercase letter. All other buttons in the same row use Title Case or Sentence case. This is the only all-lowercase button label in the interface. High Already covered in Button Labels — fix capitalisation to Disallow Reply or Restrict Reply.
Grammar Tab label: Read Message Users / Unread Message Users Awkward noun-stacking. "Read Message Users" could mean "Users who are message-readers" or "Read [this] Message Users". Ambiguous. Medium Simplify to Read and Unread as tab labels. The modal title provides the context that these are message recipients.
